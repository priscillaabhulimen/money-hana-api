# Migration Guide

This project uses SQLAlchemy models in `app/models/`, but there is currently no committed Alembic setup (`alembic/` directory and migration history are not present in the repository).

## Current State

- Runtime DB driver: `postgresql+asyncpg`
- Models are defined in:
  - `app/models/users.py`
  - `app/models/transactions.py`
  - `app/models/goals.py`
- Health endpoint: `GET /health` (checks DB connectivity)

Because migration files are not yet versioned here, schema changes are not reproducible through source control by default.

## Recommended Next Step: Add Alembic

Use this once to make schema changes trackable and shareable.

## 1. Install Migration Dependencies

```bash
pip install alembic psycopg[binary]
```

If you want this locked for teammates, update `requirements.txt` after install:

```bash
pip freeze > requirements.txt
```

## 2. Initialize Alembic

```bash
alembic init alembic
```

This creates:

- `alembic.ini`
- `alembic/`
- `alembic/env.py`
- `alembic/versions/`

## 3. Configure Alembic to Read `.env`

In `alembic/env.py`, make sure you:

1. Load `.env`
2. Read `DATABASE_URL`
3. Convert async URL to sync URL for Alembic
4. Import project models before setting `target_metadata`

Reference snippet:

```python
import os
from pathlib import Path
from dotenv import load_dotenv
from app.base import Base

from app.models.users import User  # noqa: F401
from app.models.transactions import Transaction  # noqa: F401
from app.models.goals import Goal  # noqa: F401

load_dotenv(Path(__file__).resolve().parent.parent / ".env")

database_url = os.getenv("DATABASE_URL")
if not database_url:
    raise ValueError("DATABASE_URL is not set")

sync_url = database_url.replace("postgresql+asyncpg://", "postgresql+psycopg://")
config.set_main_option("sqlalchemy.url", sync_url)

target_metadata = Base.metadata
```

## 4. Create Initial Migration

After Alembic is configured:

```bash
alembic revision --autogenerate -m "initial schema"
alembic upgrade head
```

Review generated files before applying.

## 5. Daily Workflow for Schema Changes

After model changes:

```bash
alembic revision --autogenerate -m "describe change"
alembic upgrade head
```

Useful commands:

```bash
alembic current
alembic history --verbose
alembic downgrade -1
```

## 6. Data Safety Checks (Before New Constraints)

When adding new check constraints, verify old data first. Example queries:

```sql
SELECT id, amount FROM transactions WHERE amount <= 0;
SELECT id, monthly_limit FROM goals WHERE monthly_limit <= 0;
```

Fix invalid rows before running `alembic upgrade head`.

## 7. Team Conventions

- One logical schema change per migration.
- Use clear migration messages.
- Review autogenerated migrations before apply.
- Do not rewrite old committed migrations in shared branches.
- Keep migration files in source control.

## 8. Required Change for Cookie Refresh Auth

This codebase now uses a `refresh_tokens` table for HttpOnly cookie refresh sessions.

If `APP_ENV` is not `development`, startup auto-create is disabled and this table must be created via migration (or manual DDL) before running the API.

Expected table:

- `refresh_tokens`

### Manual DDL (temporary, until Alembic is committed)

```sql
CREATE TABLE IF NOT EXISTS refresh_tokens (
  id UUID PRIMARY KEY,
  user_id UUID NOT NULL REFERENCES users (id),
  token_hash VARCHAR NOT NULL UNIQUE,
  expires_at TIMESTAMPTZ NOT NULL,
  revoked_at TIMESTAMPTZ NULL,
  created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS ix_refresh_tokens_user_id ON refresh_tokens (user_id);
CREATE INDEX IF NOT EXISTS ix_refresh_tokens_token_hash ON refresh_tokens (token_hash);
```

After creating this table, existing users can log in again to receive new refresh cookies.
